# 1. Executive Summary
This migration is not just “moving reports out of Google Sheets.” It is replacing a multi-sheet operating system used for client reporting, lead handoff, customer success management, health monitoring, finance tracking, reply automation, OOO routing, blacklist handling, and internal sales operations.

The target product is really four connected surfaces:
- Client portal
- CS manager operational workspace
- Admin portal
- Super admin management/settings area

The real source of truth is not the formal markdown spec alone. The workbooks and Make blueprints reveal materially broader requirements:
- The client report workbook defines the real client-facing reporting contract: qualified leads, campaign context, statistics, and dashboard widgets.
- The PDCA workbook defines the internal agency operating model: DoD/WoW/MoM tracking, health assessments, setup/configuration, finance, CRM, domains, exclusions, and lost-client management.
- The automation blueprints define hidden product requirements: AI reply classification, tag-driven lead logging, OOO extraction and rerouting, blacklist state, CRM and forwarding side effects, and export-to-Look4Lead workflows.

Main findings:
- The future product needs more pages and more state surfaces than the current formal spec and current prototype show.
- The current live Supabase schema is not sufficient for required UI coverage.
- The current live Supabase schema is also not permission-safe yet: RLS is enabled on tables, but there are no actual policies returned from `pg_policies`.
- The current prototype is useful for layout direction, but it is not reliable as product truth. It hardcodes role context, does not scope client data correctly, and omits several required operational modules.
- The biggest modeling problem is the lead model. Current materials require a separation between internal qualification and client-managed pipeline progression, but the live schema currently mixes stages and omits critical metadata.

Main risks:
- Building from the current live schema without changes will force frontend workarounds and break key flows.
- If role visibility is implemented from the current prototype instead of the real workbook/automation logic, clients will either see too much or miss required data.
- If reply classification and OOO handling remain “background-only,” operations will lack auditability and recovery paths.
- If the team copies the formal spec’s simplified status model, operational parity with the existing sheets system will be lost.

Major missing pieces:
- Client access mapping and invitation model
- Client daily snapshot time-series
- Health assessment history
- Structured issues tracking
- Reply classification/audit surface
- OOO review/routing surface
- Import center with validation and history
- Automation/audit logs
- Proper separation of qualification vs client pipeline stage
- Several operational fields now stored only in spreadsheet columns

Dashboard formulas show the expected client-facing widgets and aggregations.

- Role in future system: Source for OOO queue, routing management, retry/review states.

## Current demo portal `https://fluent-eagle-62420935.figma.site`
- What it contains: JS-rendered wrapper around the same code-component style demo.
- What it reveals: Confirms an existing demo surface exists, but does not add more business truth than the local prototype.
- Reliability / authority: Low for requirements; useful only as a visual reference.
- Role in future system: Demo reference only.

## Current live Supabase schema
- What it contains: Existing `public` tables with zero rows and no migrations recorded.
- What it reveals: Current implementation started but is materially incomplete. Roles, statuses, and entities do not align with the real product needs. `pg_policies` returned no policies despite RLS-enabled tables.

# 3. Current-State System Decomposition
The current system decomposes into these product capabilities.

## 3.1 Client Reporting Layer
Current capability:
- Show delivered leads
- Show campaign-level stats
- Show dashboard summaries and trends
- Let someone update downstream pipeline booleans

Implemented today via:
- Client report workbook `Leads`
- Client report workbook `Statistics`
- Client report workbook `Campaigns`
- Dashboard formulas using weekly/monthly/date-range aggregations

Operational meaning:
- The client portal is fundamentally a qualified-lead delivery and performance-reporting tool.
- The dashboard is derived, not manually authored.
- The lead list is the most operational client screen.

## 3.2 Internal Client Operations Layer
Current capability:
- Track every active client in one grid
- Monitor send schedule vs actuals
- Compare current week/month vs prior periods
- Track health states and insights
- Manage client setup and ecosystem integrations
- Track commercial/contract context

Implemented today via:
- `CS PDCA`
- `🤖Daily stats`
- `Client CRM Details`
- manual sheet columns and formulas

Operational meaning:
- The internal portal must support dense table-based analysis, not just card dashboards.

## 3.3 Lead Intake and Qualification Layer
Current capability:
- Receive Bison reply events
- Classify reply intent
- Attach tags
- Create/update actionable lead rows
- Calculate response time
- Enrich contact data
- Trigger downstream actions

Implemented today via:
- `Bison Replies Classification` blueprint
- `Log Bison Replies...` blueprint
- client `Leads` sheet
- `Classification Test`

Operational meaning:
- Internal users need a lead-processing workspace plus audit/recovery visibility for automation decisions.

## 3.4 OOO Management Layer
Current capability:
- Extract expected return date
- Store OOO leads separately
- Route by workspace and gender
- Reattach leads into follow-up campaigns

Implemented today via:
- `OOO Leads`
- `ARM`
- `Add OOO leads into Campaigns` blueprint

Operational meaning:
- OOO is a managed queue with routing logic, not a passive tag.

## 3.5 Blacklist and Compliance Layer
Current capability:
- Add/remove email or domain blacklists in Bison
- Maintain global exclude-domain list
- Persist returned blacklist IDs in lead rows

Implemented today via:
- `🤖Emails Exclude List`
- lead columns `Blacklist ID` and `Domain Blacklist ID`
- automation branches

Operational meaning:
- UI needs both global blacklist management and per-lead blacklist state visibility.

## 3.6 Internal Sales and Pipeline Layer
Current capability:
- Track GHEADS’ own sales CRM
- Track LG pipeline
- Track won/lost lessons

Implemented today via:
- `CRM`
- `LG PDCA`

Operational meaning:
- Internal sales is a separate product domain from client delivery and must remain access-isolated.

## 3.7 Finance and Commercial Tracking Layer
Current capability:
- Maintain monthly cash-flow plan
- Track client invoice state
- Track partnership payment state
- Track account-based selling scoring

Implemented today via:
- `CF`
- invoice/partnership/ABS columns in `CS PDCA`

Operational meaning:
- Finance is partially structured, partially embedded in client rows. The product should unbundle it into focused screens.

## 3.8 Access and Integration Layer
Current capability:
- Map client reports by spreadsheet ID
- Store workspace IDs and API keys
- Store CRM credentials
- Support Look4Lead export token

Implemented today via:
- `AUTH`
- `CS PDCA`
- `Client CRM Details`

Operational meaning:
- Configuration management is already a product area, even if it currently lives in columns.

## 5.4 Permission nuances that must be explicit in implementation
- Client users can update downstream pipeline state, not internal qualification.
- Client users cannot see OOO, NRR, spam, blacklist IDs, AI confidence, or raw automation decisions.
- `cs_manager` should not see raw API keys or secret tokens by default; they need connection status and masked values.
- `cs_manager` can create health assessments, update domains, manage issues, and process leads for assigned clients.
- `cs_manager` should not create/delete users, manage global integrations, or edit agency-only sales/finance.
- `admin` can manage nearly all operational data but should not rotate root-level secrets or assign/remove `super_admin` without super-admin rights.
- `super_admin` is the only role that should have impersonation, secret rotation, and high-risk system controls.
- Export permissions differ: client exports filtered own-client leads only; `cs_manager` exports assigned-client operational data; `admin` and `super_admin` export global reports.

## 6.2 Critical schema mismatches affecting frontend behavior
### Lead model mismatch
Current sources require:
- Internal qualification: `preMQL`, `MQL`
- Client-managed closing funnel: meeting scheduled, meeting held, offer sent, won, rejected
- Raw reply classification and AI metadata
- OOO and blacklist state

Live schema currently:
- Mixes qualification and pipeline into one enum
- Lacks most lead detail fields from workbook
- Has no AI classification fields on replies

Result:
- Current schema cannot cleanly support the permission model or page behavior.

Recommended UI-facing model:
- Lead record
- Reply/event history
- Separate client pipeline stage
- Separate rejection reason
- Separate raw classification metadata

### Client metrics mismatch
Current sources require:
- Daily client-level snapshots
- Weekly and monthly rollups
- Negative/OOO/human reply rates
- Schedule vs actual send comparisons

Live schema currently:
- Has `campaign_daily_stats`
- Does not have `client_daily_snapshots`
- Does not have health history
- Does not have the required derived views

Result:
- Internal PDCA pages are not feasible from the live schema alone.

### Access-control mismatch
Current sources require:
- Multiple client users per client
- Assignment of managers to subsets of clients
- Strict client-only row visibility

Live schema currently:
- Has no `client_users`
- Uses role `manager` instead of `cs_manager`
- Has no returned RLS policies

Result:
- Client-facing UI cannot be launched safely on current schema.

# 8. Client Portal Functional Specification
## Client Sign In
**Status: Missing but required**
- Route: `/login` or `/client/login`
- Primary roles: `client`
- Primary purpose: Authenticate client users and direct them into their own portal.
- Business value: Replaces spreadsheet sharing with controlled access.
- Key datasets: `users`, `client_users`, auth session, optional invitation token.
- Required fields: email; magic-link or passwordless token state; assigned client list if a user belongs to multiple clients.
- Main UI sections: sign-in form; invitation acceptance state; support/help link; branded reassurance about visible data scope.
- Filters/search/sort: none.
- Row actions: none.
- Bulk actions: none.
- Edit capabilities: email entry only; password reset if password auth is used.
- Validation rules: valid email format; expired/invalid invitation states must be handled; inactive users blocked with clear message.
- Empty states: first-time user invited but not activated yet.
- Loading states: submitting link request; verifying token.
- Error states: invalid token; inactive account; no assigned client access.
- Permission nuances: client users only; internal roles should use internal sign-in.
- Entry points: invite email; direct URL; support links.
- Exit paths / linked destinations: dashboard after auth; support contact on failure.
- Mobile/responsive considerations: fully mobile-friendly; one-column layout.
- Audit/history needs: sign-in audit event.
- Export/reporting needs: none.
- Notes / open questions: Current live schema lacks `client_users`, so multi-user client access cannot be implemented correctly yet.

## Client Dashboard
- Route: `/client/dashboard`
- Primary roles: `client`
- Primary purpose: Give the client a clear performance overview of delivered leads and campaign activity.
- Business value: Replaces the workbook dashboard and reduces the need to ask the CS team for updates.
- Key datasets: aggregated `campaign_daily_stats`; qualified `leads`; campaign summary; optionally `client_daily_snapshots` for prospect base and KPI trend cards.
- Required fields: daily sent/reply/bounce/open counts; lead counts by week and month; active campaign list with status and database size; current-period MQL/preMQL count; meetings; won; prospect base; period-over-period deltas.
- Main UI sections: KPI cards; daily activity chart; lead volume by week; lead volume by month; campaign summary table; quick links to leads and campaigns.
- Filters/search/sort: date range preset; custom date range; campaign filter; qualification scope toggle if needed.
- Row actions: open campaign detail from campaign summary.
- Bulk actions: none.

# 11. Automation-Driven UI Requirements
## 11.1 Reply classification
Must be visible:
- raw incoming reply
- `automated_reply` flag
- AI category
- confidence
- short reason / reasoning
- detected language
- attached Bison tag
- whether lead row was created, updated, or ignored

Must be editable/reviewable:
- manual reclassification
- approval of low-confidence results
- re-run processing
- “ignore” or “not actionable” resolution

Must be traceable:
- timestamp
- client/workspace
- source webhook event id if available
- downstream side effects

## 11.2 OOO handling
Must be visible:
- expected return date
- source reply
- gender used for routing
- workspace and resolved route campaign
- attach-to-campaign outcome
- processed/unprocessed state

Must be editable:
- expected return date override
- gender override
- route override
- attach retry

Must be traceable:
- every attach attempt
- failure reason
- whether processed flag changed

## 11.3 Blacklist management
Must be visible:
- whether a lead/email/domain was blacklisted
- blacklist IDs written by automation
- source lead/reply that triggered blacklisting
- removal events

Must be editable:
- add/remove blacklist manually from internal pages
- add/remove domain in global exclude list

Must be traceable:
- actor or automation
- timestamp
- target type: email vs domain
- external provider response

## 11.4 Lead enrichment and forwarding
Must be visible:
- whether Snov/Lusha enrichment succeeded
- phone source
- whether SMS was sent
- whether reply was forwarded and to whom

Must be traceable:
- provider used
- provider result
- fallback path used

## 11.5 CRM push and external exports

- Look4Lead export status per lead
- last attempt and error message
- destination integration used

Must be triggerable:
- retry CRM push
- retry Look4Lead export

## 11.6 Import and scheduled-job visibility
Must be visible:
- import job history
- automation run history
- success/failure status
- affected record counts
- downloadable error summary

# 16. Risks, Ambiguities, and Product Decisions Needed
1. Lead visibility
Explicit: workbook shows both `preMQL` and `MQL`.
Inferred: clients currently rely on both.
Ambiguous: alternate spec text says only `MQL` should be visible.
Recommended decision: expose both `preMQL` and `MQL` to clients for parity.

2. Lead-state model
Explicit: workbook has `QUALIFICATION` plus separate meeting/offer/won fields.
Explicit: current live DB mixes stages into one enum.
Recommended decision: split internal qualification from client pipeline stage.

3. Client status taxonomy
Explicit: workbook uses `Active`, `Abo`, `On hold`, `Offboarding`, `Inactive`, `Sales`.
Explicit: formal spec proposes different statuses.
Recommended decision: use workbook statuses in MVP and optionally map them to grouped display labels.

4. Reply taxonomy
Explicit: automations use `OOO`, `Interested`, `NRR`, `Spam_Inbound`, `Left_Company`, `other`.
Explicit: formal spec uses a different set.
Recommended decision: maintain an internal automation classification enum distinct from lead qualification.

5. OOO timing rule
Explicit: one blueprint attaches on `now - 2 days`; spec text implies `<= current_date`.
Recommended decision: use a processed flag plus `expected_return_date <= current_date` to avoid silent misses.

6. `SQL` terminology
Explicit: PDCA workbook uses “SQL leads received”.
Inferred: in practice this does not map cleanly to a separate final status.
Recommended decision: retire ambiguous `SQL` labels from UI unless stakeholders define them precisely.

7. CRM configuration shape
Explicit: workbook stores platform + keys + secret + subdomain.
Ambiguous: whether multiple CRM connections per client are needed.
Recommended decision: build one active CRM integration per client in MVP, but do not block a future 1:N model.

8. Finance scope for CS managers
Explicit: formal matrix gives managers finance visibility for assigned clients.
Ambiguous: whether they can edit invoice state or only view it.
Recommended decision: managers may view invoice status and add operational notes only; admin retains financial control.

9. E5M-related sheets
Explicit: `E5M CS`, `Prospect Base`, `Weekly`, `Monthly` depend on those structures.
Ambiguous: whether these belong in the GHEADS product scope.
Recommended decision: confirm with stakeholders; treat as non-MVP unless clearly still in active GHEADS ops.

10. Current schema access safety
Explicit: RLS enabled; `pg_policies` returned no policies.
Recommended decision: treat this as a launch blocker for any client-facing release.
